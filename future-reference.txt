Day 2: PostgreSQL – Schema Design & Normalization

Awesome! Day 2 is where the relational magic happens. Today, we are building the structural foundation of LitCritic in PostgreSQL.

Relational databases thrive on normalization—organizing data to reduce redundancy and improve data integrity. Since a single book can have multiple genres, and a genre can belong to many books, we need a "many-to-many" relationship.

Let’s dive into the SQL.

🛠️ Step 1: Open psql Inside Your Docker Container
To run our SQL commands, we need to access the PostgreSQL command-line interface (psql) running inside the Docker container we created yesterday.

Open your terminal and run:

Bash
docker exec -it postgres-lit psql -U postgres -d postgres
(You should see your prompt change to postgres=#, meaning you are inside the database!)

🏗️ Step 2: The Schema Design (DDL)
Copy and paste the following SQL script directly into your psql prompt and hit Enter. This will create our 5 normalized tables.

SQL
-- 1. Users Table
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Authors Table
CREATE TABLE authors (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    bio TEXT
);

-- 3. Genres Table
CREATE TABLE genres (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL
);

-- 4. Books Table (One-to-Many: An author can have many books)
CREATE TABLE books (
    id SERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    author_id INTEGER REFERENCES authors(id) ON DELETE CASCADE,
    published_year NUMERIC(4, 0),
    summary TEXT
);

-- 5. Book_Genres Table (Many-to-Many: Books <-> Genres)
CREATE TABLE book_genres (
    book_id INTEGER REFERENCES books(id) ON DELETE CASCADE,
    genre_id INTEGER REFERENCES genres(id) ON DELETE CASCADE,
    PRIMARY KEY (book_id, genre_id)
);

🧠 Day 2 Concept Checklist: What We Just Recalled
3NF Normalization: Notice how books doesn't contain the author's name or the genre name directly as text. We separated them into authors and genres tables to prevent duplication. If an author changes their name, we only update it in one place.

Foreign Keys & Constraints: REFERENCES authors(id) ensures we can't add a book for an author that doesn't exist. ON DELETE CASCADE means if we delete an author, all their books are automatically deleted too (no orphaned data!).

Composite Keys: In the book_genres table, PRIMARY KEY (book_id, genre_id) ensures that the exact same book and genre combination cannot be linked twice. The combination of both IDs is the unique identifier.

Data Types: We utilized SERIAL for auto-incrementing IDs, VARCHAR for short strings, TEXT for longer flexible strings, and NUMERIC(4,0) to ensure the year is exactly a 4-digit number without decimals.

You now have a fully normalized, seeded relational database! Take a moment to verify your data if you'd like (you can run SELECT * FROM books; inside psql). When you're ready to start building our Express routes to fetch and join this data, just reply with "Day 3"!

📖 Step 2: GET All Books (The JOIN Masterclass)
To get a book, its author, and its multiple genres in a single API call, we need to join 4 different tables. We will also use Postgres's incredibly powerful json_agg function to bundle the many-to-many genres into a clean JavaScript array.

Add this route to your index.js (inside startServer, below the /ping route):

JavaScript
    // GET /books - Fetch all books with authors and genres
    app.get('/books', async (req, res) => {
      try {
        const query = `
          SELECT 
            b.id, 
            b.title, 
            b.published_year, 
            a.name AS author,
            COALESCE(json_agg(g.name) FILTER (WHERE g.name IS NOT NULL), '[]') AS genres
          FROM books b
          INNER JOIN authors a ON b.author_id = a.id
          LEFT JOIN book_genres bg ON b.id = bg.book_id
          LEFT JOIN genres g ON bg.genre_id = g.id
          GROUP BY b.id, a.name
          ORDER BY b.id ASC;
        `;
        const result = await pgClient.query(query);
        res.status(200).json(result.rows);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to fetch books" });
      }
    });
🎯 Step 3: GET a Book by ID (Parameterized Queries)
Never trust user input! Instead of concatenating strings (which causes SQL injection), we use $1 to let the pg driver safely inject the variable.

Add this route:

JavaScript
    // GET /books/:id - Fetch a single book
    app.get('/books/:id', async (req, res) => {
      try {
        const bookId = req.params.id;
        const query = `
          SELECT b.id, b.title, b.published_year, b.summary, a.name AS author
          FROM books b
          INNER JOIN authors a ON b.author_id = a.id
          WHERE b.id = $1;
        `;
        const result = await pgClient.query(query, [bookId]); // $1 is replaced by bookId

        if (result.rows.length === 0) {
          return res.status(404).json({ error: "Book not found" });
        }
        res.status(200).json(result.rows[0]);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to fetch book" });
      }
    });
🛡️ Step 4: POST a New Book (Transactions!)
Creating a book with genres means we must insert data into the books table AND the book_genres table. If the first insert succeeds but the second fails, we get corrupted data. Transactions (BEGIN and COMMIT) ensure it's "all or nothing."

Add this route:

JavaScript
    // POST /books - Create a new book with genres (Using a Transaction)
    app.post('/books', async (req, res) => {
      const { title, author_id, published_year, genre_ids } = req.body;

      try {
        // 1. Start Transaction
        await pgClient.query('BEGIN');

        // 2. Insert into Books table and return the new ID
        const insertBookQuery = `
          INSERT INTO books (title, author_id, published_year) 
          VALUES ($1, $2, $3) RETURNING id;
        `;
        const bookResult = await pgClient.query(insertBookQuery, [title, author_id, published_year]);
        const newBookId = bookResult.rows[0].id;

        // 3. Insert into Book_Genres join table (if genres were provided)
        if (genre_ids && genre_ids.length > 0) {
          const insertGenreQuery = `
            INSERT INTO book_genres (book_id, genre_id) VALUES ($1, $2);
          `;
          for (const genreId of genre_ids) {
            await pgClient.query(insertGenreQuery, [newBookId, genreId]);
          }
        }

        // 4. Commit Transaction (Save permanently)
        await pgClient.query('COMMIT');

        res.status(201).json({ success: true, message: "Book created!", bookId: newBookId });
      } catch (err) {
        // 5. Rollback Transaction on error (Undo everything)
        await pgClient.query('ROLLBACK');
        console.error(err);
        res.status(500).json({ error: "Transaction failed. No data was saved." });
      }
    });

🗄️ Step 1: Database Updates (SQL)
Open pgAdmin's Query Tool (or your Docker psql terminal) and run this script. It will add a popularity counter, create our mock reviews, seed some data, and—most importantly—build our Indexes and Materialized View.

SQL
-- 1. Add a popularity counter to our books
ALTER TABLE books ADD COLUMN views_count INTEGER DEFAULT 0;

-- Give our existing books some random views so we have data to sort
UPDATE books SET views_count = floor(random() * 1000 + 1);

-- 2. Create a temporary mock_reviews table
CREATE TABLE mock_reviews (
    id SERIAL PRIMARY KEY,
    book_id INTEGER REFERENCES books(id) ON DELETE CASCADE,
    rating NUMERIC(2,1) CHECK (rating >= 1 AND rating <= 5),
    review_text TEXT
);

-- Insert some fake ratings for Dune, Dune Messiah, and 1984
INSERT INTO mock_reviews (book_id, rating) VALUES
(1, 4.5), (1, 5.0), (1, 4.0),
(2, 3.5), (2, 4.0),
(3, 5.0), (3, 4.8), (3, 4.9);

-- 3. OPTIMIZATION: Create Indexes
-- Indexes act like the index at the back of a book, drastically speeding up searches
CREATE INDEX idx_books_title ON books(title);
CREATE INDEX idx_authors_name ON authors(name);

-- 4. OPTIMIZATION: Create a Materialized View for Analytics
-- This runs a heavy calculation ONCE and saves the result as a physical table
CREATE MATERIALIZED VIEW mv_book_ratings AS
SELECT 
    b.id AS book_id,
    b.title,
    ROUND(AVG(r.rating), 2) AS avg_rating,
    COUNT(r.id) AS review_count
FROM books b
LEFT JOIN mock_reviews r ON b.id = r.book_id
GROUP BY b.id, b.title;

-- Required to allow us to refresh the view concurrently later
CREATE UNIQUE INDEX idx_mv_book_ratings_id ON mv_book_ratings(book_id);

    // ---------------------------------------------------------
    // DAY 4: ADVANCED QUERIES & OPTIMIZATION
    // ---------------------------------------------------------

    // 1. GET /search - Search across title, author, and genre
    app.get('/search', async (req, res) => {
      try {
        const { q } = req.query; // Extracts ?q=something from the URL
        if (!q) return res.status(400).json({ error: "Missing search query 'q'" });

        // We use % for wildcard matching (e.g., %dune% matches "Children of Dune")
        const searchQuery = `%${q}%`;
        
        // ILIKE is Postgres-specific for Case-Insensitive matching
        const query = `
          SELECT DISTINCT b.id, b.title, a.name AS author
          FROM books b
          INNER JOIN authors a ON b.author_id = a.id
          LEFT JOIN book_genres bg ON b.id = bg.book_id
          LEFT JOIN genres g ON bg.genre_id = g.id
          WHERE b.title ILIKE $1 
             OR a.name ILIKE $1 
             OR g.name ILIKE $1;
        `;
        const result = await pgClient.query(query, [searchQuery]);
        res.status(200).json(result.rows);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Search failed" });
      }
    });

    // 2. GET /analytics/top-books - Fast sorting using our new column
    app.get('/analytics/top-books', async (req, res) => {
      try {
        const query = `
          SELECT id, title, views_count 
          FROM books 
          ORDER BY views_count DESC 
          LIMIT 10;
        `;
        const result = await pgClient.query(query);
        res.status(200).json(result.rows);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to fetch top books" });
      }
    });

    // 3. GET /analytics/ratings - Fetching from the Materialized View!

🧠 Day 4 Concept Checklist:
GROUP BY & Aggregates: In our SQL script, we used AVG() and COUNT(). You must use GROUP BY when mixing aggregate functions with normal columns (like title), so Postgres knows how to group the math.

Indexes: By adding an Index on title, Postgres creates a highly optimized B-Tree under the hood. When millions of users search for a book, Postgres no longer has to scan every single row one by one (Sequential Scan); it jumps straight to the answer (Index Scan).

Materialized Views: Standard Views run the query every single time they are called. A Materialized View runs the query once and saves the actual data to the disk. It trades a tiny bit of storage space for massive read performance. (Note: If new reviews are added, you have to run REFRESH MATERIALIZED VIEW mv_book_ratings; to update it!).

EXPLAIN ANALYZE: Want to see the indexes in action? Go to pgAdmin's query tool and run EXPLAIN ANALYZE SELECT * FROM books WHERE title = 'Dune';. It will spit out a report showing exactly how many milliseconds the database took to plan and execute the query!

Day 5: PostgreSQL – Transactions, Functions & Triggers

You are officially at the halfway mark! Day 5 is where we separate the basic CRUD apps from enterprise-grade software. Today is all about ACID properties, concurrency, and pushing logic down into the database itself.

When money or inventory is involved (like buying a book), we cannot risk the Node.js server crashing halfway through a request and leaving our data corrupted. We also want to automate repetitive tasks directly in PostgreSQL.

Let’s add inventory control, triggers, and stored functions to LitCritic.

🗄️ Step 1: Database Updates (Functions & Triggers)
First, we need to add stock and last_updated columns to our books, and create an orders table. Then, we will write a trigger to automatically update that timestamp, and a function to calculate a dynamic popularity score.

Open pgAdmin's Query Tool (or your Docker psql terminal) and run this SQL script:

SQL
-- 1. Add Stock and Timestamp columns
ALTER TABLE books ADD COLUMN stock INTEGER DEFAULT 10;
ALTER TABLE books ADD COLUMN last_updated TIMESTAMP DEFAULT CURRENT_TIMESTAMP;

-- 2. Create an Orders table
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    book_id INTEGER REFERENCES books(id) ON DELETE CASCADE,
    user_id INTEGER, -- We will use a fake integer for now (e.g., User 99)
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 3. THE TRIGGER FUNCTION: What happens when a book is updated?
CREATE OR REPLACE FUNCTION update_modified_column()
RETURNS TRIGGER AS $$
BEGIN
    -- 'NEW' holds the new row data being updated
    NEW.last_updated = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 4. THE TRIGGER: When does the function run?
CREATE TRIGGER set_timestamp
BEFORE UPDATE ON books
FOR EACH ROW
EXECUTE FUNCTION update_modified_column();

-- 5. THE STORED FUNCTION: Calculate dynamic popularity
-- Score = Views + (Total Orders * 50)
CREATE OR REPLACE FUNCTION get_book_popularity(b_id INTEGER)
RETURNS INTEGER AS $$
DECLARE
    v_views INTEGER;
    v_orders INTEGER;
    total_score INTEGER;
BEGIN
    -- Fetch current views
    SELECT views_count INTO v_views FROM books WHERE id = b_id;
    -- Fetch total orders for this book
    SELECT COUNT(*) INTO v_orders FROM orders WHERE book_id = b_id;
    
    -- Calculate and return the score
    total_score := COALESCE(v_views, 0) + (COALESCE(v_orders, 0) * 50);
    RETURN total_score;
END;
$$ LANGUAGE plpgsql;
💻 Step 2: The Express Routes (index.js)
Now, let's implement the "Buy Book" endpoint. We will use a strict Transaction with a Row-Level Lock (FOR UPDATE). This guarantees that if two users try to buy the last copy of Dune at the exact same millisecond, only one will succeed.

Add these two routes to your index.js file:

JavaScript
    // ---------------------------------------------------------
    // DAY 5: TRANSACTIONS, TRIGGERS & FUNCTIONS
    // ---------------------------------------------------------

    // 1. POST /orders - Buy a book (ACID Transaction)
    app.post('/orders', async (req, res) => {
      const { book_id, user_id } = req.body;

      try {
        await pgClient.query('BEGIN'); // Start Transaction

        // Step A: Check stock and LOCK THE ROW so no other transaction can modify it
        const checkStockQuery = 'SELECT stock FROM books WHERE id = $1 FOR UPDATE';
        const stockResult = await pgClient.query(checkStockQuery, [book_id]);

        if (stockResult.rows.length === 0) {
          await pgClient.query('ROLLBACK');
          return res.status(404).json({ error: "Book not found" });
        }

        const currentStock = stockResult.rows[0].stock;
        
        // Step B: Application-level consistency check
        if (currentStock <= 0) {
          await pgClient.query('ROLLBACK');
          return res.status(400).json({ error: "Sorry, this book is out of stock!" });
        }

        // Step C: Reduce stock (Our PostgreSQL Trigger will auto-update 'last_updated' here!)
        const updateStockQuery = 'UPDATE books SET stock = stock - 1 WHERE id = $1';
        await pgClient.query(updateStockQuery, [book_id]);

        // Step D: Record the order
        const insertOrderQuery = 'INSERT INTO orders (book_id, user_id) VALUES ($1, $2) RETURNING id';
        const orderResult = await pgClient.query(insertOrderQuery, [book_id, user_id]);

        await pgClient.query('COMMIT'); // Commit Transaction (Save permanently)

        res.status(201).json({
          success: true,
          message: "Order placed successfully!",
          orderId: orderResult.rows[0].id
        });

      } catch (err) {
        await pgClient.query('ROLLBACK'); // Undo everything on error
        console.error(err);
        res.status(500).json({ error: "Transaction failed. Order cancelled." });
      }
    });

    // 2. GET /books/:id/popularity - Execute our Stored Function
    app.get('/books/:id/popularity', async (req, res) => {
      try {
        // We call the function directly in the SELECT statement
        const query = 'SELECT get_book_popularity($1) AS score';
        const result = await pgClient.query(query, [req.params.id]);
        
        res.status(200).json({ book_id: req.params.id, popularity_score: result.rows[0].score });
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to calculate popularity" });
      }
    });

🧠 Day 5 Concept Checklist:
ACID in Action: * Atomicity: Both the stock reduction and order creation succeed, or neither do (BEGIN / COMMIT / ROLLBACK).

Consistency: Our code prevents stock from dropping below zero.

Isolation: FOR UPDATE locks the book row. If 100 users try to buy it at once, Postgres forces them into a queue, preventing race conditions.

Durability: Once committed, the order is safe even if the Docker container loses power a second later.

Triggers vs Application Logic: Instead of writing last_updated: new Date() in our Node.js UPDATE query, we gave that job to the database. This guarantees the timestamp updates no matter who or what modifies the table (whether it's our Node app, Python script, or a manual pgAdmin edit).

Stored Functions (plpgsql): We encapsulated the complex math of "Popularity" safely inside the database engine, reducing the amount of data we have to send over the network.

This is a brilliant question. When you start mixing application code (Node.js) with database logic (Triggers/Functions), it can get a little hazy figuring out exactly who is doing what, and in what order.

Let's break down the exact timeline of a single request hitting our POST /orders API.

Here is the step-by-step journey of that data, from the moment you click "Send" in Postman:

🌊 The Execution Flow: POST /orders
1. The Client Trigger (Postman)
You send an HTTP POST request with { "book_id": 1, "user_id": 99 } to http://localhost:3000/orders.

2. The Express Route (Node.js)
Your Node.js server receives the request, parses the JSON body, and enters the app.post('/orders') route.

3. The Transaction Begins (Node.js ➡️ PostgreSQL)
Node.js sends the BEGIN command to Postgres. A protective "bubble" is created. Nothing inside this bubble is permanent until we say so.

4. The Row Lock (Node.js ➡️ PostgreSQL)
Node.js sends: SELECT stock FROM books WHERE id = 1 FOR UPDATE;

Behind the scenes: Postgres finds the row for Dune and puts a padlock on it. If another user tries to buy Dune at this exact millisecond, Postgres forces them to wait.

5. The Application Check (Node.js)
Node.js looks at the stock number it just received. Is it greater than 0? Yes. The flow continues.

6. The Update Query (Node.js ➡️ PostgreSQL)
Node.js sends: UPDATE books SET stock = stock - 1 WHERE id = 1;

7. ⚡ THE TRIGGER FIRES (Inside PostgreSQL)
Stop right here! Before Postgres actually saves that update to the hard drive, its internal alarm goes off.

Postgres sees we created a BEFORE UPDATE ON books trigger named set_timestamp.

Postgres automatically halts the update process and executes the attached function: update_modified_column().

The Function's Job: It looks at the incoming data (NEW), grabs the last_updated column, and forces it to equal the current clock time.

Now Postgres allows the update to finish, saving both the new stock number AND the new timestamp. (Node.js had no idea this happened!).

8. The Insert Query (Node.js ➡️ PostgreSQL)
Node.js sends: INSERT INTO orders (book_id, user_id) VALUES (1, 99);

9. The Commit (Node.js ➡️ PostgreSQL)
Node.js sends the COMMIT command.

Postgres officially saves everything to the disk.

Postgres unlocks the Dune row so other waiting users can now buy it.

10. The Response (Node.js ➡️ Client)
Node.js packages up a success message and sends an HTTP 201 Created status back to Postman.

🤔 Wait, what about the Popularity Function?
You might notice that our stored function get_book_popularity() was not triggered during the purchase flow!

Triggers happen automatically in the background based on database events (like an UPDATE).

Functions only run when explicitly called.

Our popularity function only executes when Node.js explicitly asks for it by sending SELECT get_book_popularity(1); via the GET /books/:id/popularity endpoint.

Day 6: MongoDB – Document Model & Flexible Schema

Welcome to Day 6! Today is a major paradigm shift. You are leaving the strict, tabular world of SQL and entering the flexible, document-based world of NoSQL.

Let’s quickly refresh the core concepts before we write some code:

BSON & Flexible Schema: MongoDB stores data in BSON (Binary JSON). It looks like JSON to us, but it's optimized for speed and supports extra data types (like Dates and ObjectIds). Unlike SQL tables, MongoDB collections don't enforce a rigid schema by default. Two documents in the same collection can have entirely different fields.

Embedded Documents vs. References: This is the most important MongoDB design concept.

References (like SQL foreign keys) are used when data grows infinitely (e.g., a user's purchase history).

Embedding is used for data that is frequently read together and doesn't grow infinitely. In our case, embedding comments directly inside the review document is perfect because when we fetch a review, we almost always want to see its comments immediately.

_id ObjectId: Every document gets a unique _id automatically. It’s not just a random string; an ObjectId is a 12-byte identifier that actually contains a timestamp of when it was created!
