Day 2: PostgreSQL – Schema Design & Normalization

Awesome! Day 2 is where the relational magic happens. Today, we are building the structural foundation of LitCritic in PostgreSQL.

Relational databases thrive on normalization—organizing data to reduce redundancy and improve data integrity. Since a single book can have multiple genres, and a genre can belong to many books, we need a "many-to-many" relationship.

Let’s dive into the SQL.

🛠️ Step 1: Open psql Inside Your Docker Container
To run our SQL commands, we need to access the PostgreSQL command-line interface (psql) running inside the Docker container we created yesterday.

Open your terminal and run:

Bash
docker exec -it postgres-lit psql -U postgres -d postgres
(You should see your prompt change to postgres=#, meaning you are inside the database!)

🏗️ Step 2: The Schema Design (DDL)
Copy and paste the following SQL script directly into your psql prompt and hit Enter. This will create our 5 normalized tables.

SQL
-- 1. Users Table
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Authors Table
CREATE TABLE authors (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    bio TEXT
);

-- 3. Genres Table
CREATE TABLE genres (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL
);

-- 4. Books Table (One-to-Many: An author can have many books)
CREATE TABLE books (
    id SERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    author_id INTEGER REFERENCES authors(id) ON DELETE CASCADE,
    published_year NUMERIC(4, 0),
    summary TEXT
);

-- 5. Book_Genres Table (Many-to-Many: Books <-> Genres)
CREATE TABLE book_genres (
    book_id INTEGER REFERENCES books(id) ON DELETE CASCADE,
    genre_id INTEGER REFERENCES genres(id) ON DELETE CASCADE,
    PRIMARY KEY (book_id, genre_id)
);

🧠 Day 2 Concept Checklist: What We Just Recalled
3NF Normalization: Notice how books doesn't contain the author's name or the genre name directly as text. We separated them into authors and genres tables to prevent duplication. If an author changes their name, we only update it in one place.

Foreign Keys & Constraints: REFERENCES authors(id) ensures we can't add a book for an author that doesn't exist. ON DELETE CASCADE means if we delete an author, all their books are automatically deleted too (no orphaned data!).

Composite Keys: In the book_genres table, PRIMARY KEY (book_id, genre_id) ensures that the exact same book and genre combination cannot be linked twice. The combination of both IDs is the unique identifier.

Data Types: We utilized SERIAL for auto-incrementing IDs, VARCHAR for short strings, TEXT for longer flexible strings, and NUMERIC(4,0) to ensure the year is exactly a 4-digit number without decimals.

You now have a fully normalized, seeded relational database! Take a moment to verify your data if you'd like (you can run SELECT * FROM books; inside psql). When you're ready to start building our Express routes to fetch and join this data, just reply with "Day 3"!

📖 Step 2: GET All Books (The JOIN Masterclass)
To get a book, its author, and its multiple genres in a single API call, we need to join 4 different tables. We will also use Postgres's incredibly powerful json_agg function to bundle the many-to-many genres into a clean JavaScript array.

Add this route to your index.js (inside startServer, below the /ping route):

JavaScript
    // GET /books - Fetch all books with authors and genres
    app.get('/books', async (req, res) => {
      try {
        const query = `
          SELECT 
            b.id, 
            b.title, 
            b.published_year, 
            a.name AS author,
            COALESCE(json_agg(g.name) FILTER (WHERE g.name IS NOT NULL), '[]') AS genres
          FROM books b
          INNER JOIN authors a ON b.author_id = a.id
          LEFT JOIN book_genres bg ON b.id = bg.book_id
          LEFT JOIN genres g ON bg.genre_id = g.id
          GROUP BY b.id, a.name
          ORDER BY b.id ASC;
        `;
        const result = await pgClient.query(query);
        res.status(200).json(result.rows);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to fetch books" });
      }
    });
🎯 Step 3: GET a Book by ID (Parameterized Queries)
Never trust user input! Instead of concatenating strings (which causes SQL injection), we use $1 to let the pg driver safely inject the variable.

Add this route:

JavaScript
    // GET /books/:id - Fetch a single book
    app.get('/books/:id', async (req, res) => {
      try {
        const bookId = req.params.id;
        const query = `
          SELECT b.id, b.title, b.published_year, b.summary, a.name AS author
          FROM books b
          INNER JOIN authors a ON b.author_id = a.id
          WHERE b.id = $1;
        `;
        const result = await pgClient.query(query, [bookId]); // $1 is replaced by bookId

        if (result.rows.length === 0) {
          return res.status(404).json({ error: "Book not found" });
        }
        res.status(200).json(result.rows[0]);
      } catch (err) {
        console.error(err);
        res.status(500).json({ error: "Failed to fetch book" });
      }
    });
🛡️ Step 4: POST a New Book (Transactions!)
Creating a book with genres means we must insert data into the books table AND the book_genres table. If the first insert succeeds but the second fails, we get corrupted data. Transactions (BEGIN and COMMIT) ensure it's "all or nothing."

Add this route:

JavaScript
    // POST /books - Create a new book with genres (Using a Transaction)
    app.post('/books', async (req, res) => {
      const { title, author_id, published_year, genre_ids } = req.body;

      try {
        // 1. Start Transaction
        await pgClient.query('BEGIN');

        // 2. Insert into Books table and return the new ID
        const insertBookQuery = `
          INSERT INTO books (title, author_id, published_year) 
          VALUES ($1, $2, $3) RETURNING id;
        `;
        const bookResult = await pgClient.query(insertBookQuery, [title, author_id, published_year]);
        const newBookId = bookResult.rows[0].id;

        // 3. Insert into Book_Genres join table (if genres were provided)
        if (genre_ids && genre_ids.length > 0) {
          const insertGenreQuery = `
            INSERT INTO book_genres (book_id, genre_id) VALUES ($1, $2);
          `;
          for (const genreId of genre_ids) {
            await pgClient.query(insertGenreQuery, [newBookId, genreId]);
          }
        }

        // 4. Commit Transaction (Save permanently)
        await pgClient.query('COMMIT');

        res.status(201).json({ success: true, message: "Book created!", bookId: newBookId });
      } catch (err) {
        // 5. Rollback Transaction on error (Undo everything)
        await pgClient.query('ROLLBACK');
        console.error(err);
        res.status(500).json({ error: "Transaction failed. No data was saved." });
      }
    });

🧠 Day 3 Concept Checklist:
INNER vs LEFT JOIN: We used INNER JOIN for authors because every book must have an author. We used LEFT JOIN for genres because theoretically, a book might not have a genre assigned yet, and we still want the book to show up in our list!

Parameterized Queries ($1, $2): This separates the SQL logic from the user data, making SQL injection attacks practically impossible.

Transactions: BEGIN, COMMIT, and ROLLBACK guarantee our database remains in a valid state, enforcing the "Atomicity" part of ACID.
